# 벤치마크가 입증한 것 — 목적·결과·신뢰의 근거

> 본 문서는 [REPORT.md](./REPORT.md)의 정량 결과를 비즈니스·신뢰 관점에서 해석합니다.
> 데이터: [results.json](../public/benchmark/results.json), 대시보드: http://localhost:3000/benchmark

---

## 1. 왜 테스트를 했나

LexOS는 **AI 기본법(2026.1.22 시행)에 익숙하지 않은 스타트업 대표**가 자사 코드/프로필을 입력하면 어떤 의무가 트리거되는지, 위반 시 무엇이 문제인지를 30초 안에 보여주는 도구입니다.

이 도구는 **딱 두 가지 약속** 위에 서 있습니다:

1. **틀린 법령을 만들어내지 않는다 (no hallucination)** — AI 응답에 등장하는 모든 조문은 RAG corpus에서 가져온 실재 텍스트여야 한다.
2. **컴플라이언스 의무를 빠뜨리지 않는다 (no false negative on vuln)** — 명백히 위험한 코드는 명백히 의무를 트리거해야 한다.

이 두 약속이 실제로 지켜지는지를 **외부에서 검증 가능한 형태**로 확인하는 것이 본 테스트의 목적입니다.

---

## 2. 어떻게 검증 가능하게 만들었나

직접 만든 10개 GitHub 레포에 **각각 어떤 의무가 트리거되어야 하는지를 사전에 라벨링**한 뒤(취약 5 / 안전 5), 같은 LexOS 파이프라인에 모두 통과시켰습니다.

| 분류 | 의도된 트리거 | 레포 |
|---|---|---|
| 취약 | 신용 자동심사 → HIGH-IMPACT, FOREIGN-REP | CMUX-test01 |
| 취약 | 의료 진단 자동화 → HIGH-IMPACT, NOTICE | CMUX-test02 |
| 취약 | 생성 이미지 + 워터마크 부재 → WATERMARK 위반 | CMUX-test03 |
| 취약 | 채용 자동결정 → HIGH-IMPACT, employment | CMUX-test04 |
| 취약 | LLM 파인튜닝 + PII 컬럼 → DATA-GOVERNANCE | CMUX-test05 |
| 안전 | HyperCLOVA 사내 CLI → 외산 대리인 의무 회피 | CMUX-test06 |
| 안전 | 전통 ML 추천 → 비-생성형, 저위험 | CMUX-test07 |
| 안전 | 생성 이미지 + 워터마크 코드 존재 → WATERMARK 의무 이행 | CMUX-test08 |
| 안전 | 한국 LLM 챗봇 + AI 고지문 → NOTICE 의무 이행 | CMUX-test09 |
| 안전 | AI 미사용 Express CRUD → 의무 0개 | CMUX-test10 |

이 라벨링이 곧 **정답지**입니다. LexOS가 각 레포를 어떻게 판정하는지가 정답과 얼마나 일치하는지를 본 것입니다.

## 3. 결과로 입증된 것

### ✅ 약속 ① — 환각 차단이 실제로 작동한다

| 지표 | 값 | 의미 |
|---|---:|---|
| RAG 인용 검증 통과율 | **42 / 42 = 100%** | 모든 적용 의무 항목이 corpus 발췌와 substring 일치하는 인용을 동반 |
| 본문 환각 조문 (`unsupportedRefs`) | **0 건** | reasoning 본문이 corpus 외 조문 번호를 만들어낸 사례 없음 |
| 알 수 없는 의무 ID 드롭 | **0 건** | 9개 화이트리스트 외 ID 등장 없음 |

이건 단순히 "AI가 잘 작동했다"가 아니라 **"AI가 거짓말을 해도 시스템이 잡아냈을 것"** 을 의미합니다. 검증 로직 자체가 결정론적(deterministic) substring 매칭이므로, 통과한 모든 인용은 사후에 누구든 corpus 파일과 대조해 같은 결과를 재현할 수 있습니다.

### ✅ 약속 ② — 위험 코드는 위험하다고 말한다

| 취약 레포 4건 (test01 truncation 제외) | 위험 등급 | 적용 의무 수 |
|---|---|---:|
| 의료 영상 (test02) | high | 7 |
| 이미지 생성 워터마크X (test03) | high | 4 |
| 채용 스크리닝 (test04) | high | 6 |
| PII 파인튜닝 (test05) | high | 4 |

**4/4 모두 high로 판정**, 의도한 핵심 의무(고영향·고지·워터마크·데이터 거버넌스)가 트리거됨.

### ✅ 안전 분류도 분리됐다

- AI 미사용 (test10) → 의무 0개, low
- 워터마크 이행 (test08) → 워터마크 violation 회피 (안전 패턴 인식)
- 한국 LLM (test06, test09) → AIBA-FOREIGN-REP 미발동

## 4. 신뢰의 근거를 한 줄로

> **"틀리면 보이고, 맞으면 추적 가능하다."**

- **틀리면 보인다**: test01은 응답 truncation으로 리포트가 실패했고, UI는 이를 빨간 에러로 표시합니다 — 조용히 잘못된 답을 내놓지 않습니다.
- **맞으면 추적 가능하다**: 9개 성공한 리포트에서 의무 42건 모두 인용 조문이 어느 corpus 항목인지(`verifiedLocator`) 라벨링되어 있어, 법무팀이 "이 주장의 근거는 무엇이냐"고 물으면 즉시 출처를 보여줄 수 있습니다.

이 두 속성은 일반 LLM 응답에는 없는 것이고, 그게 LexOS가 ChatGPT에 "AI 기본법 알려줘"라고 묻는 것과 다른 점입니다.

## 5. 본 테스트가 입증하지 *못한* 것 (정직하게)

1. **법적 자문 수준의 정확성** — 라벨링은 코드 패턴 기반이며 변호사 검토가 아닙니다. "high impact"가 곧 "법적 위반"은 아닙니다.
2. **LLM 결정성** — 단일 실행이며 같은 입력에 대해 ±10% 정도 변동이 있을 수 있습니다. 안정성은 반복 실행으로 별도 측정 필요.
3. **9개 외 의무** — RAG corpus는 9개만 다룹니다. 하위 시행령·시행규칙은 corpus 추가 후에야 다룰 수 있습니다.
4. **악의적 입력** — 저장소 자체가 prompt-injection 공격을 시도하면 어떻게 될지는 별도 테스트 필요.

또한 발견된 **결함 2건**은 정직하게 공개합니다:

- **test01 truncation** — 토큰 한도(16k)가 의무 9개 + 시스템 6개를 한 번에 담기에 부족했음. 의무 단위 분할 호출로 수정 예정.
- **test07 false positive** — `pd.read_csv` + `score > threshold` 패턴이 도메인 키워드 없이 트리거되어 안전한 추천 시스템이 high로 분류됨. 룰 정밀화 필요.

---

## 6. 결론

LexOS의 핵심 차별점인 **"환각 없는 컴플라이언스 진단"** 은 본 벤치마크에서 **9개 성공 사례 전부에서 인용 검증과 환각 차단이 작동**함으로써 경험적으로 뒷받침됐습니다. 의무 분류의 false positive 1건과 응답 truncation 1건은 알려진 한계이며, 둘 다 코드 수준에서 수정 가능한 결함입니다.

스타트업 대표가 이 도구를 신뢰할 수 있는 이유는 결과를 믿어달라고 부탁하기 때문이 아니라, **모든 결과의 출처를 그 자리에서 corpus 파일과 대조해 검증할 수 있기 때문**입니다.
